## 🎯 基本方針

### 最重要原則

**すべてのユーザー向け出力は日本語で行う**

内部的な思考プロセスは英語でも構いませんが、ユーザーに提示するすべての内容は**必ず日本語に翻訳**してください。

> ⚠️ **重要**: 英語で考えた内容をそのまま出力してはいけません。必ず日本語に変換してから出力してください。

---

## 言語使用の原則

### ✅ 日本語で出力すべきもの

以下はすべて**日本語**で出力してください:

1. **説明文・応答メッセージ**
   - ユーザーへの返答
   - 状況説明
   - 質問や確認

2. **コメント・ドキュメント**
   - コード内のコメント
   - ドキュメント文字列（docstring）
   - README、ガイド等のドキュメント

3. **ユーザー向けメッセージ**
   - エラーメッセージ（カスタムメッセージ）
   - ログメッセージ（ユーザー向け）
   - コンソール出力（ユーザー向け）

4. **提案・推奨事項**
   - コードレビューコメント
   - 改善提案
   - ベストプラクティスの説明

5. **ドキュメント更新の提案**
   - 💡 の提案文
   - プレビュー内容
   - 更新理由の説明

### ❌ 英語のまま保持すべきもの

以下は**英語**のまま保持してください:

1. **コード要素**
   - 変数名（例: `userName`, `fileCount`）
   - 関数名（例: `fetchData()`, `processItems()`）
   - クラス名（例: `UserService`, `DataProcessor`）
   - メソッド名
   - プロパティ名

2. **技術的識別子**
   - APIエンドポイント（例: `/api/users`）
   - 環境変数名（例: `NODE_ENV`）
   - データベース名・テーブル名
   - ファイルパス（例: `src/components/Button.tsx`）

3. **ライブラリ・フレームワーク由来**
   - ライブラリが提供するエラーメッセージ
   - フレームワークの組み込み関数名
   - 外部APIのレスポンスメッセージ

4. **Git関連**
   - コミットメッセージ（プロジェクト方針に従う）
   - ブランチ名
   - タグ名

## 重要な注意点

### 思考プロセスと出力の分離

| フェーズ | 言語 | 説明 |
|---------|------|------|
| 内部思考 | 英語可 | Claude内部の推論・分析は英語でも問題なし |
| 出力準備 | 翻訳必須 | ユーザーに見せる前に必ず日本語に翻訳 |
| 最終出力 | 日本語 | ユーザーが見るものはすべて日本語 |

### ⚠️ よくある間違い

#### 間違い1: 英語の思考をそのまま出力

**問題:**
内部で英語で考えた内容を、翻訳せずにそのまま出力してしまう

**例:**
```
Let me read the configuration file first...
```

**修正:**
```
まず設定ファイルを読み込みます...
```

**原則**: 内部思考は英語でも構いませんが、出力前に必ず日本語に翻訳してください。

---

#### 間違い2: ドキュメント更新提案が英語

**問題:**
💡 ドキュメント更新の提案が英語で表示される

**例:**
```
💡 Document Update Proposal

Found: Error resolution for npm install failure
Should we record this in troubleshooting.md?
```

**修正:**
```
💡 ドキュメント更新の提案

発見内容: npm install 失敗時のエラー解決方法
troubleshooting.md に記録しますか?
```

**原則**: プロジェクト固有の提案フォーマットも必ず日本語で表示してください。

## 例外ケース

### 例外1: 引用・参照

外部の文献やライブラリのドキュメントを引用する場合、原文が英語であればそのまま引用し、その後に日本語の説明を追加してください。

**例:**
```
公式ドキュメントには以下のように記載されています:

> "This function returns a Promise that resolves when all operations complete."

つまり、この関数はすべての操作が完了したときに解決される Promise を返します。
```

### 例外2: ログメッセージ（開発者向け）

システムログや開発者向けのデバッグメッセージで、ライブラリが出力する英語のログをそのまま記録する場合は英語のままで問題ありません。

**例:**
```javascript
// ✅ 許容される
console.error('Database connection failed:', error);

// ただし、ユーザー向けメッセージは日本語
const userMessage = 'データベースへの接続に失敗しました';
```

### 例外3: コミットメッセージ

プロジェクトのコミットメッセージのルールに従ってください。プロジェクト方針で英語が指定されている場合は英語を使用します。

出力前に以下を確認してください:

- [ ] ユーザーへの説明文は日本語になっているか
- [ ] コメントは日本語で書かれているか
- [ ] ドキュメント文字列は日本語になっているか
- [ ] コード要素（変数名、関数名等）は英語のままか
- [ ] ユーザー向けメッセージは日本語になっているか
- [ ] 英語で考えた内容を翻訳し忘れていないか

- ✅ 一貫性のあるドキュメント
- ✅ 国際的な開発標準との互換性
